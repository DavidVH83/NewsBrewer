I Built the Same AI Tool Twice: GitHub Actions vs Claude Code Routines

A real-world test of two automation approaches for an AI newsletter digest. Here's what actually happened, including what broke.

Every day I receive a flood of AI newsletters across Gmail and Outlook. Most of it I never read. I wanted a tool that would do the reading for me, filtering what's relevant, summarizing what matters, and delivering a clean daily digest to my inbox.

So I built NewsBrewer, an open-source AI newsletter digest tool. Then I built it twice: once with GitHub Actions, once with Claude Code Routines, Anthropic's brand-new cloud automation feature launched April 14, 2026.

This is a hands-on comparison. Not a feature overview, a real test with real failures.

"The biggest surprise wasn't what failed. It was why it failed, and what that reveals about how Routines thinks."

What NewsBrewer Does

NewsBrewer connects to Gmail via IMAP, reads newsletter emails from the last 24 hours, fetches full article content, uses AI to filter what's relevant for an AI developer, and sends a formatted HTML digest to your inbox. It also picks up links you manually forward with "BREW:" in the subject line.

# The daily pipeline
Gmail IMAP  →  Extract URLs  →  Fetch articles
    →  AI filter + summarize  →  Send digest email

The tool was built using Claude Code in about 45 minutes: one prompt, six agents working autonomously. But that's a separate story. This article is about what happened when I tried to run the same automation two different ways.

Version 1
GitHub Actions

Python code running on GitHub's free CI/CD infrastructure, scheduled via cron. Full SMTP access. Zero cost.

Version 2
Claude Code Routines

Anthropic's new cloud automation. Configure a prompt once, run on schedule. No code deployment needed.

Version 1: GitHub Actions

Setup

The GitHub Actions approach requires actual code. Claude Code built the full Python pipeline autonomously from a detailed plan document. Setup involved creating Gmail and Outlook App Passwords, adding them as GitHub Secrets, and configuring a cron workflow.

# .github/workflows/daily_brew.yml
on:
  schedule:
    - cron: '0 6 * * *'  # 06:00 UTC daily
  workflow_dispatch:    # Manual trigger for testing

Total setup time: approximately 45 minutes, including the Claude Code build session. Technical level required: medium. You need to understand GitHub, environment variables, and Python basics.

Results after 3 days

✅
Gmail reading via IMAP

Connects cleanly using App Password authentication. Filters by sender, extracts URLs from both HTML and plain-text email bodies.

✅
Email delivery via SMTP

Sends formatted HTML digest to Outlook inbox every morning. Reliable, no issues after 3 consecutive runs.

✅
AI filtering and summarization

GitHub Models (Llama 4 Scout) filters by relevance, writes 2-3 sentence summaries, flags courses separately. Works well at zero API cost.

A note on article fetching

Some newsletter platforms block direct HTTP fetches with a 403 error, regardless of which automation approach you use. This is a general web scraping limitation, not specific to GitHub Actions or Routines. The fix is straightforward: use r.jina.ai/[URL] as a reader proxy, or switch to RSS feeds where available. Both approaches handle this the same way.

Version 2: Claude Code Routines

What are Routines?

Routines are Claude Code's new cloud automation feature. You configure a prompt, a repository, and connectors once, then run on a schedule, via API call, or in response to GitHub events. Nothing depends on your laptop being open.

The appeal for NewsBrewer was obvious: no YAML, no deployment, no GitHub Secrets management. Just describe what you want and set a schedule.

Run 1: Routines defaults to building, not running

I gave Routines a prompt describing what NewsBrewer should do: read emails, summarize articles, send a digest. The prompt described the system without explicitly telling it to execute the task right now.

What happened next surprised me.

"Claude Code Routines didn't try to run the task. It built the entire application and pushed it to GitHub."

Instead of executing the digest workflow, the Routine interpreted my prompt as a development task. It wrote a complete Python application with five modules, committed everything to a new branch called claude/elegant-edison-vxPqZ, and opened a pull request.

This is not a bug. It's a fundamental insight about how Routines work: Claude Code thinks like a developer by default. Give it an ambiguous prompt in a coding context, and it will write code, not run a workflow.

# What the Routine created on Run 1
newsbrewer/
  gmail_fetcher.py     # IMAP → fetch emails (last 24h)
  url_extractor.py     # Extract article URLs
  article_fetcher.py   # Fetch & parse content
  ai_processor.py      # Filter + summarize
  digest_sender.py     # Send HTML email via SMTP
Committed to: claude/elegant-edison-vxPqZ
Pull request: opened ✓

The fix: explicit execution instructions

The solution is simple once you understand the root cause. Add explicit instructions to prevent code-writing behavior:

# Essential additions to any execution-focused Routine prompt
Do not create files.
Do not use git.
Do not write code.
Just read, summarize, and send the email.
Report what you found and whether the email was sent.

Run 2: The Gmail connector creates drafts, not sends

With the corrected prompt, the Routine executed correctly: read Gmail, found 15 articles, filtered to 6 relevant ones, wrote summaries. Then it tried to send the email.

It couldn't.

The Gmail connector provides gmail_create_draft but not gmail_send_draft. This is documented in Anthropic's support center:

# From Anthropic's official documentation
"The send function is not enabled — all emails must be
 sent manually through your Gmail account."

# Available tools in Gmail connector
gmail_search_messages    ✓
gmail_read_message       ✓
gmail_create_draft       ✓
gmail_send_draft         ✗  # Does not exist
gmail_send_email         ✗  # Does not exist

This is a deliberate security decision by Anthropic. Automatically sending emails on behalf of a user is an irreversible action. The digest was saved as a ready-to-send draft in my Gmail inbox instead.

Run 3: The SendGrid workaround and why it failed

I tried to work around the sending limitation using the SendGrid API, a standard email delivery service with a free tier of 100 emails per day. The idea: the Routine reads Gmail, builds the digest, then calls SendGrid's API to deliver it.

Storing the API key securely was straightforward. Routines has a built-in custom environment for secrets, similar to GitHub Secrets. The key was stored as SENDGRID_API_KEY in the Routine's environment configuration.

The call failed with a clear error:

403 Host not in allowlist
"outbound HTTPS to api.sendgrid.com is blocked
 by the sandbox network policy"

Claude Code Routines run in a sandboxed cloud environment. Outbound HTTPS to external APIs is blocked by network policy. This isn't a SendGrid configuration issue. It's the Routines execution environment itself.

Three Things I Learned

1. Routines defaults to developer mode

If your prompt could be interpreted as a software specification, Routines will build the software. This is powerful for code-generation workflows, and a footgun for execution workflows. Always add explicit "do not write code" instructions when you want Routines to act, not build.

2. The sandbox is more restricted than expected

Routines can read from connected services via connectors, but outbound HTTPS to arbitrary external APIs is blocked. This makes sense from a security perspective: autonomous agents calling external APIs without review is genuinely risky. But it limits what Routines can do today as a general automation platform.

3. The Gmail connector is read-only by design

The inability to send email is not a gap waiting to be filled. It's a deliberate choice. Anthropic requires explicit user action to send emails. For a newsletter digest that's meant to run unattended, this is a fundamental mismatch with the use case.

When to Use Which

Use GitHub Actions when

You need full autonomy and control

Email delivery, external API calls, open-source sharing, no daily limits, and zero cost make GitHub Actions the right choice for production automations that run unattended. The setup is more technical, but you own the entire pipeline.

Use Routines when

You want to prototype fast with connected services

Routines shine for workflows that live entirely within Claude's connector ecosystem — summarizing Slack threads, triaging GitHub issues, analyzing Drive documents. If your automation doesn't need to send email or call external APIs, Routines is dramatically faster to set up and iterate on.

Conclusion

Routines is a genuinely exciting tool. The setup experience is dramatically simpler than GitHub Actions, and for the right use cases (code review automation, issue triage, document analysis), it will be transformative.

But for email automation specifically, it's not ready yet. The combination of a read-only Gmail connector and a sandboxed network environment means fully autonomous email delivery isn't possible today.

For NewsBrewer, GitHub Actions remains the right choice. It works, it's free, and it gives full control over the pipeline.

"Routines is Claude Code's most ambitious feature yet. The sandbox limitations are real, but they're also version 1.0 on day one of a research preview."

I'll be watching the connector capabilities closely. The moment Gmail gets a send tool, the Routines version becomes far more compelling.
